## 0. 项目综述 / 自我介绍类

### Q0.1 用一分钟介绍这个项目
> 这是一个 provider 无关的**多 Agent 创意生成编排平台**。用户输入一句创意，系统走一条固定的多阶段流水线——目标锁定 → RAG 检索 → 提示词工程 → 能力预检 → 调用生成模型 → 质量评审 → 交付——生成图片或带配音的短视频，并支持连续多轮对话创作。技术上我做了三件核心的事：①把 LLM 的各子任务封装成 Agent、串成可观测可复算的链路；②真实的 RAG 检索（pgvector 向量 + 关键词混合 + rerank）；③provider 抽象，让底层模型（DashScope/Sora/Gemini）可插拔。后端 Java17/SpringBoot、DDD 分层，前端 Vue3，数据落 PostgreSQL+pgvector。

**要点**：先说"是什么"，再说"我做的三件核心事"，最后一句技术栈。不要一上来堆细节。

### Q0.2 这个项目最大的技术挑战是什么？
> 三个：①**LLM 输出不可控**——自由文本没法喂给下游程序，我用结构化输出（JSON schema + 容错解析 + 重试）+ 质量门禁解决；②**生成有随机性导致偶发失败**——同样的输入有时缺字段、评分不达标，我做了 harness 重试循环（同阶段重试、换备用凭证）把成功率从经常失败提到 ~4/5；③**持久化与增量落库的一致性**——链路异步逐阶段落库，切到 PostgreSQL 后暴露了外键顺序 bug，我用调整保存顺序 + 软化外键迁移解决。

### Q0.3 系统架构讲一下（白板题）
> 四层 DDD：`interfaces`(REST) → `application`(用例编排，链路引擎在这) → `domain`(纯领域模型，不依赖框架) ← `infrastructure`(provider 网关、仓储、RAG 实现，实现 domain 的接口)。核心是 `ChainRunExecutionService`：按 `StageCatalog` 逐阶段执行，每阶段完成即落库，异步推进，前端轮询。外部依赖 PostgreSQL+pgvector、DashScope/Pollinations 模型、ffmpeg。

**画图顺序**：先画四层框，再画链路 I00→I60，再画 provider 路由分叉到 DashScope/Pollinations/Fixture。

### Q0.4 你在项目里具体负责什么 / 哪些是你设计的？（AI 辅助如何回答）
> 诚实但有力：整体架构分层、链路阶段划分、provider 抽象、质量门禁与重试策略、RAG 融合方案、上下文压缩策略都是我设计的；实现过程中借助了 AI 辅助编码，但每一处设计取舍、踩的坑和修复都是我主导的——比如评分算出 101 撑爆 DB 约束、外键顺序导致链路崩、末帧接力做人物一致性这些，都是我定位并解决的。

## 1. Agent 开发 / 大模型应用

### Q1.1 什么是 Agent？你项目里的 Agent 怎么设计的？
> Agent 是让 LLM «扮演指定角色、按指定格式产出» 完成一个明确子任务的封装。我的每个阶段由一个或多个 Agent 节点组成，比如目标锁定阶段的 GoalAgent 把用户口语化输入规整成结构化目标（scene/style 等字段），提示词阶段的 PromptAgent 产出正/负向提示词包。每个 Agent = 一段 system prompt（角色+输出契约）+ 结构化输出解析 + 该阶段的评审规则。

### Q1.2 你这是多 Agent 系统吗？为什么这么拆？串行还是并行？
> 是。拆分原则是**单一职责 + 可独立评审**：每个 Agent 只干一件事，产出可以被单独打分/重试，出问题能定位到具体阶段。**主链路是串行**（后一步依赖前一步产出，如提示词依赖目标）；但在"剧情扩展"这类**处理量大、维度独立**的场景我用了**同层 Agent 并行**——一个 agent 管环境、一个管人物、一个管特效……用线程池并发，实测 5~7 个 agent 并行比串行快 ~3.4x。**并行的前提是各子任务无依赖**，有依赖的必须串行。

### Q1.3 你的编排和 ReAct / Plan-and-Execute / Function Calling 有什么区别？
> - **ReAct** 是"思考→行动→观察"动态循环，模型自己决定下一步调什么工具；
> - **Plan-and-Execute** 是先让模型出计划再逐步执行；
> - **Function Calling** 是模型输出结构化的工具调用意图。
>
> 我的项目是**预定义的固定工作流（DAG/pipeline）**，不是让模型动态决策路径——因为创意生成的流程是确定的（目标→检索→提示词→生成→评审），固定流水线**更可控、可观测、可复算、好加质量门禁**，也更省 token。代价是灵活性低。如果要做"用户任意任务"的通用 Agent，才需要 ReAct 这种动态规划。**能说清这个取舍是加分点。**

### Q1.4 怎么保证 LLM 输出可控？JSON 解析失败怎么办？
> 三层：①**结构化输出**——system prompt 里给 JSON schema，明确"只输出 JSON、字段含义、约束"，必要时给 few-shot 示例；②**容错解析**——很多模型会输出带 markdown 包裹或被截断的 JSON，我做了容错解析：剥 ```json``` 包裹、用栈匹配找最外层 `{}`（string-aware，不被字符串内花括号干扰）、截断时尽量闭合救回；③**重试 + 校验**——解析失败或关键字段缺失就重试（LLM 有随机性，重跑常能过），配合下游的字段非空校验（如 scene/style must not be blank）。

### Q1.5 大模型幻觉/输出质量不稳怎么处理？
> ①**质量门禁**：`StageReviewPolicy` 按固定 rubric 给产物打分（0-100），图片看视觉/目标匹配/一致性，视频还接 qwen-vl 视觉模型**真看图打分**，不达标打回；②**harness 重试**：不达标同阶段重生成、复审，仍不过换备用凭证；③**RAG 接地**：让生成参考检索到的真实上下文，减少凭空编造；④**结构化约束**缩小输出空间。关键是**别信模型一次就对，要有门禁 + 重试闭环**。

### Q1.6 上下文窗口 / token 超了怎么办？（你的多轮记忆压缩）
> 我做了**阈值 + 语义压缩**：同一对话的多轮请求累积，超过上下文阈值（可按模型 context 预算配）时，**最近 N 轮保留原文（verbatim），更早的旧轮用 LLM 语义压缩成一句"已确立设定"**（只保留会影响后续生成的关键信息：主体/风格/角色），LLM 不可用则回退启发式保住"最初设定"锚点。核心思想：**近期上下文和最初锚点这两处最不能丢，中间可归纳**。比无脑截断（可能丢主体）和全量塞（爆 token）都好。

**追问：为什么不用向量记忆？** 可以升级为向量化会话记忆（把历史存向量、按相关性召回），适合超长对话；我当前场景对话不长，阈值+压缩性价比更高，且实现简单、延迟低。

### Q1.7 Prompt 工程你怎么做的？
> ①**角色 + 输出契约**放 system；②**结构化 schema**约束输出；③**关键约束前置、具体化**（如"只输出 JSON、第一个字符是 {、禁止解释"）；④必要时 **few-shot**；⑤**温度**按任务调（结构化/摘要用低温 0.3，创意用高温）；⑥**幂等的确定性注入**——比如人物一致性，我不靠模型每次记住，而是把"固定外貌描述"用代码逐字注入每个镜头的 prompt，保证一致。**最后一条最能体现工程思维：能用代码保证的别交给模型的随机性。**

### Q1.8 Agent 的可观测性 / 怎么调试？
> 每阶段完成即落库（stage_run + review_report + external_job + agent_node_run），前端轮询能看到**逐阶段进度、每步的评审分数、调了哪个 provider job**。调试时能精确定位到哪个阶段、哪个 agent、评审为什么不过。这也是我选**固定流水线**而非动态 ReAct 的原因之一——可观测性好太多。

### Q1.9 怎么评估 Agent / 生成质量？
> ①**过程质量**：每阶段的 rubric 评审分；②**检索质量**：RAG 有独立评测 harness（Recall@K/Precision@K/MRR）；③**产物质量**：接 qwen-vl 视觉模型对图/视频真打分（人物一致性、元素覆盖），比如末帧接力后人物一致性从 60 提到 98。生产化还应建**固定评测集 + 回归基线**，每次改动跑一遍防退化——这是我 roadmap 里要补的。

### Q1.10 换模型（Gemini/Sora/Imagen）要改什么？
> **几乎不用改业务**。我做了 provider 抽象：`RoutingProviderHttpGateway` 按 `capabilityType + provider` 名路由到具体网关，能力通过"能力配置页"运行时录入。接 Sora = 写一个 `SoraVideoProviderHttpGateway`（实现 `supportsCapability` + `invoke`）+ 路由注册 + 前端配 key 启用，链路/评审/前端一行不改。**这就是抽象的价值：把"用哪个模型"和"业务逻辑"解耦。**

### Q1.11 成本怎么控制？
> ①分层选模型：文本规划用便宜的 qwen-turbo/plus，只有最终渲染才用贵的图/视频模型；②免费兜底：Pollinations/Fixture 做开发和降级；③token 压缩（多轮记忆）；④缓存/幂等避免重复生成；⑤能力槽让用户自己配 key、自己承担成本。项目里我一直有成本意识（文本设计全流程 ~¥0.02）。

## 2. RAG / 大模型开发

### Q2.1 什么是 RAG？为什么用？
> Retrieval-Augmented Generation：生成前先从知识库/历史检索相关内容，作为上下文喂给 LLM。目的：①注入模型不知道的私有/最新知识；②减少幻觉（有依据地生成）；③省 token（只给相关片段而非全量）。我的项目用它检索历史创作/设定，让生成延续既有风格与一致性。

### Q2.2 向量检索原理？embedding？相似度？
> 把文本用 **embedding 模型**映射成高维向量（语义相近→向量相近），检索时把 query 也向量化，找最近邻。相似度常用**余弦相似度**（cos θ = A·B/(|A||B|)，只看方向不看模长，适合文本）。也有内积、L2 距离。我用 DashScope text-embedding，存 pgvector。

### Q2.3 pgvector 怎么用？索引？
> pgvector 给 PostgreSQL 加 `vector` 类型和距离运算符（`<->` L2、`<=>` 余弦、`<#>` 内积）。数据量大时建近似索引：**HNSW**（图索引，查询快、召回高、内存大）或 **IVFFlat**（倒排+聚类，建索引快、内存小）。小数据可暴力精确扫。选 pgvector 的原因：**复用 PostgreSQL**，不用额外引入 Milvus/Faiss，运维简单、能和业务数据一个事务。

**追问：pgvector vs 专用向量库？** 数据量在千万级以内、想少运维一个组件 → pgvector；亿级、超高 QPS、需要更强 ANN → Milvus/专用库。

### Q2.4 你的混合检索 + RRF 是什么？为什么比单路好？
> 我同时跑**向量检索**（语义相近）和**关键词检索**（字面重叠），再用 **RRF（Reciprocal Rank Fusion）** 融合两路排名。RRF 公式：一个文档的融合分 = Σ 1/(k + rank_i)（k 常取 60，rank 是它在各路的名次）。好处：向量擅长语义、关键词擅长精确匹配（专有名词/ID），**融合后互补**——纯向量会漏掉字面精确但语义偏的，纯关键词会漏掉换了说法的。RRF 不需要调各路分数量纲，只按排名，鲁棒。

### Q2.5 rerank 是什么？为什么要？
> 检索召回的是"粗排"（快但不够准），rerank 用更强的模型（cross-encoder，query 和 doc 一起进模型算相关性）对 top-N 精排。因为召回追求快+高召回，精度不够；rerank 追求准，但慢，只对少量候选做。两阶段"召回→精排"是检索系统标准范式。

### Q2.6 chunking（切块）策略？
> 文档要切成合适粒度再向量化：太大→一个块混多个主题、检索不准；太小→丢上下文。常见按段落/句子/固定 token 窗口切，带 overlap（重叠）避免切断语义。要按内容类型调。

### Q2.7 检索质量怎么评估？Recall@K / Precision@K / MRR 定义？
> - **Recall@K**：前 K 个结果里，命中的相关文档 / 全部相关文档。衡量"该找到的找回了多少"。
> - **Precision@K**：前 K 个里相关的比例。衡量"找回的有多少是对的"。
> - **MRR（Mean Reciprocal Rank）**：第一个正确结果排名的倒数的平均（1/rank）。衡量"正确答案排得多靠前"。
>
> 我建了评测 harness 跑这几个指标 + namespace 污染检查，防止检索退化。

### Q2.8 embedding 模型怎么选？维度？
> 看语言（中文选支持中文的）、维度（高维更细但更贵/慢）、领域（通用 vs 领域微调）、成本。维度决定存储和检索开销。选定后**入库和查询必须同一个模型**，否则向量空间对不上。

## 3. 后端开发 / 系统设计

### Q3.1 你的链路引擎怎么设计的？
> `ChainRunExecutionService` 是核心：`while(index < stages)` 逐阶段 `buildStage`，每阶段做「写 RAG 上下文 → 检索 → 执行 agent 节点 → 协调合并产出 → 评审 → 记录产出」，**完成即落库**。设计上把"阶段定义"(`StageCatalog`)、"执行"、"评审"(`StageReviewPolicy`)、"协调"(`StageCoordinator`) 解耦，好扩展、好测。

### Q3.2 为什么异步？怎么实现的？
> 生成很慢（图几秒、视频几十秒到几分钟），HTTP 同步会超时、阻塞线程。所以 `start` 立即返回 `EXECUTING`，把执行**提交到线程池**后台跑，逐阶段落库，**前端轮询** `GET /chain-runs/{id}` 看进度。测试环境用同步开关（`chain-run.sync=true`）方便断言。

**追问：为什么不用 WebSocket/SSE 推送？** 轮询实现简单、无状态、易扩展；生产化可升级为 SSE 推进度减少轮询开销。

### Q3.3 数据库设计？讲讲那个外键的坑（高频，很能体现深度）
> 表：project / chain_run / stage_run / review_report / artifact / external_job / agent_node_run / 知识库相关。**坑**：external_job、agent_node_run 在 agent 执行期间就写入、引用 stage_run，但此时 stage_run 还没落库；而且 chain_run 保存用"删除后重插 stage_run"，配 `ON DELETE CASCADE` 会在每次增量保存时连带清空这些子表。这套增量写入模型是按内存仓储设计的，切到 PostgreSQL 硬外键就冲突了。**我的解法**：①先存 chain_run 再存快照（修保存顺序）；②Flyway 迁移 V3 把这几个子表对 stage_run 的**硬外键改成软引用**（应用层保证一致性）。**这题能讲清楚，面试官会认为你真做过生产。**

### Q3.4 增量落库的一致性怎么保证？
> 每阶段是一个相对独立的写入单元，落库后前端能立即看到。跨阶段没用一个大事务，因为①链路长、大事务持锁久、②增量可见性（前端要看进度）比强一致更重要。一致性靠**应用层**保证（状态机推进、软引用）。这是**为可观测性和长流程做的一致性取舍**。

### Q3.5 并发问题？多个链路同时跑？
> 每个链路在独立线程执行，用独立的 chainRunId 命名空间，产物按 chain_run 隔离，互不干扰。共享的仓储用线程安全实现（内存版 `ConcurrentHashMap`）。要注意的是同一链路不能被并发重复推进——用状态判断（是否已取消/终态）控制。

### Q3.6 幂等性？重试怎么保证不出乱子？
> 重试的是"生成+评审"这种**可重复执行**的阶段——每次重试产出新产物、新评审，不依赖上次的中间状态（失败阶段不 record 输出，所以重试是干净的）。外部 provider 调用有 traceId、external_job 记录，便于排查重复。真正的分布式幂等（如防重复扣费）本项目没到那步，但能说清"哪些操作幂等、为什么"。

### Q3.7 链路状态流转（状态机）？
> chain_run 有 EXECUTING / SUCCEEDED / FAILED / CANCELLED / WAITING_CAPABILITY / WAITING_REVIEW 等状态。质量不达标→重试→换凭证→都不行则收敛为软阻塞态（可再次生成，不是硬崩）。前端按状态显示不同 UI。**把它讲成一个状态机**，面试官爱听。

### Q3.8 可观测性（日志/监控）？
> 现状：结构化日志（关键阶段、重试都打日志）、每步落库可回溯、全局异常处理器统一兜底（且**必须打日志**——我踩过异常被吞没法排查的坑）。生产化要补：Micrometer 指标（各阶段耗时、成功率、provider 延迟）、链路追踪（traceId 串起来）、告警。这是我明确知道的短板。

### Q3.9 媒体文件存储方案？
> DashScope 返回的是**会过期的 OSS 预签名链接**，历史产物会失效。我做了 `MediaRehoster`：下载转存本地 `storage/media`，返回永久 `/media/**` URL（Spring 静态资源托管）。生产化应上**对象存储 + CDN**（本地不扛并发和容量）。这题体现你考虑了"链接过期"这种真实生产问题。

### Q3.10 如果 QPS 上来了怎么扩展？（系统设计重点）
> ①**无状态化 + 水平扩展**：后端多实例，状态全落库/缓存；②**任务队列**：生成任务进 MQ（Kafka/RabbitMQ），worker 池消费，削峰填谷、失败重投；③**读写分离 + 缓存**：热点读走 Redis/只读库；④**向量检索**加 HNSW 索引 / 独立向量库；⑤**媒体走对象存储 + CDN**；⑥**限流 + 熔断**（provider 挂了要熔断降级）；⑦**幂等 + 去重**防重复生成。核心思路：**长耗时任务异步化 + 队列 + 无状态水平扩展**。

### Q3.11 provider 挂了怎么办？
> ①能力槽支持配多个 key/provider，评分不达标或调用失败时**换备用凭证/provider**；②免费兜底（Pollinations/Fixture）降级；③生产化加**熔断**（连续失败快速失败、不拖垮）、**超时 + 重试（指数退避）**——我在配音那块踩过 429 限流，就是用退避重试解决的。

### Q3.12 API 设计（RESTful）？
> 资源化路径（`/projects/{id}/image-chain-runs`）、动作用子资源或 `:verb`（`:redo`/`:select`）、统一响应包 `{success,data,error}`、状态用 HTTP code + 业务 code。异步资源用"创建即返回 + 轮询"模式。

### Q3.13 安全怎么做的？
> ①**密钥 AES 加密入库**（`ApiKeyProtector`），前端只见脱敏值，明文只提交一次；密钥文件不入库不提交；②CORS 白名单；③敏感配置走环境变量注入，不写进提交的配置文件；④全局异常处理**不把 SQL/堆栈暴露给用户**（我修过这个——原来直接把 SQL 报错甩到前端）。

## 4. Java 语言与框架

### Q4.1 为什么用 DDD / 六边形架构？分层依赖方向？
> 为了**领域逻辑与技术细节解耦**。`domain` 层放纯业务模型和仓储/网关**接口**，不依赖任何框架（可单测、可移植）；`infrastructure` 实现这些接口（Postgres 仓储、HTTP 网关）；`application` 编排用例；`interfaces` 暴露 REST。依赖方向：`interfaces→application→domain←infrastructure`，**依赖倒置**——高层不依赖低层实现，都依赖 domain 的抽象。好处：换数据库/换 provider 不动 domain。

### Q4.2 Spring 的 IoC/DI 讲一下？你项目怎么用的？
> IoC=控制反转，对象的创建和依赖装配交给容器；DI=依赖注入，容器把依赖注进来。我用构造器注入（不可变、好测）。`RoutingProviderHttpGateway` 就是把多个 provider 网关注入进来按需路由。

### Q4.3 @Profile / @Value / 自动配置？
> `@Profile("postgres")` 让仓储在不同 profile 切换实现（内存 vs Postgres），启动带 `--spring.profiles.active=postgres` 就用持久化版；`@Value` 注入配置（如 `aimv.postgres.url`）；自动配置是 Spring Boot 按 classpath/条件自动装配 Bean。我用 `@Profile` 实现了"测试用内存库、生产用 Postgres"同一套代码两种仓储。

### Q4.4 Bean 生命周期 / 单例？
> 默认单例，容器启动时创建、依赖注入、（可选）`@PostConstruct` 初始化、销毁时 `@PreDestroy`。单例 Bean 要注意线程安全（我共享状态用并发容器）。

### Q4.5 并发：线程池 / CompletableFuture / 并发容器？
> ①链路异步执行用**线程池**（`TaskExecutor`）提交后台任务；②多 Agent 并行扩展用线程池并发跑、汇总结果；③共享的内存仓储用 `ConcurrentHashMap` 保证线程安全。要点：**线程池要控大小**（避免 provider 并发超限，我踩过 i2v 并发被拒、TTS 429 的坑，用限并发 + 退避解决）。

### Q4.6 你项目里用了哪些 Java 新特性？
> `record`（DTO、值对象，不可变）、`Optional`（仓储 findById 返回，避免 null）、Stream（集合处理/过滤映射）、文本块（`"""`，写 SQL/JSON schema）、`var`、switch 增强。体现对现代 Java 的掌握。

### Q4.7 全局异常处理怎么做？踩过什么坑？
> `@RestControllerAdvice` + `@ExceptionHandler` 统一兜底，返回统一错误包，**不暴露技术细节给用户**。坑：原来的全局处理器**吞掉异常不打日志**，导致线上"系统处理失败"却查不到根因——我加了 `log.error` 才定位到是外键/评分越界。**教训：兜底可以，但必须记录，否则等于黑盒。**

### Q4.8 你用了哪些设计模式？
> ①**适配器模式**：各 `XxxProviderHttpGateway` 把不同厂商 API 适配成统一接口；②**策略/路由**：`RoutingProviderHttpGateway` 按能力+provider 选策略；③**工厂**：`DefaultAgentNodeFactory` 造 agent 节点；④**模板方法**：链路各阶段统一骨架、差异化实现。能把代码里的模式说出名字+为什么用，很加分。

### Q4.9（可能问）JVM / GC / 内存模型？
> 若问：堆/栈/方法区，对象在堆、引用在栈；GC 分代（新生代 Minor GC、老年代 Full GC），常见 G1；内存模型 JMM 保证可见性（volatile）、有序性、原子性。**这块和项目关联弱，按八股准备即可，别硬扯项目。**

## 5. 场景深挖 / 开放题

### Q5.1 让你支持真正的长视频/短剧（1 分钟连贯）怎么设计？
> 现状 `wanx i2v` 只能出 ~5s 片段，我用**末帧接力**（上一段末帧作下一段输入）拼接 + 分角色配音 + 音画对齐 + 真评审，跨镜头人物一致性做到 98。**根本升级**是换 **Sora/Veo** 出长镜头（消掉拼接），但跨镜头人物锁定（参考图）、口型（Wav2Lip）、配音、评审门禁这套编排仍要保留——**模型只是最后一环，编排才是系统**。这题正好呼应"是不是模型限制"：视频连贯性确实是模型瓶颈，但"像一部剧"靠编排。

### Q5.2 怎么保证生成质量稳定？
> 见质量门禁 + harness 重试 + RAG 接地 + 确定性注入（能用代码保证的不交给模型随机性）。核心闭环：**生成→评审→不达标重试→仍不行降级/换凭证**。

### Q5.3 项目有什么不足？你会怎么改进？
> 诚实列：①视频模型是瓶颈（换 Sora）；②单机、缺分布式和可观测（加 MQ/指标/追踪）；③多 Agent 剧情扩展还在原型没落进主链路；④评估体系要建回归基线；⑤媒体该上对象存储+CDN。**能主动、有条理地讲不足，比假装完美更让面试官信任。**

### Q5.4 如果让你重做，哪里会不一样？
> 一开始就按 Postgres 持久化设计（避免内存模型和外键的冲突）；provider 抽象更早引入统一的重试/熔断/限流中间层；异步进度用 SSE 而非轮询；一开始就建评测集。

## 6. 你遇到的最难的 Bug（高频必问，准备 3 个讲透）

### Bug 1：评分算出 101 撑爆 DB 约束，前端看到原始 SQL
> **现象**：视频生成失败，前端直接显示一大段 `INSERT INTO review_report ... violates check constraint`。**定位**：`overall_score` 有 `CHECK BETWEEN 0 AND 100`，而视频评审权重加起来是 **1.05**（0.20×3+0.15×2+0.10+0.05），满分算出 105/101 越界。**修复**：①权重修正为合计 1.0；②加 `clampScore` 一律钳制 0-100（防御性）；③前端 FAILED 只显示"生成失败"，不暴露 SQL。**收获**：既要修根因（权重）也要加防御（钳制），还要修 UX（不泄露技术细节）。

### Bug 2：切 PostgreSQL 后链路一启动就崩（外键）
> 见 Q3.3。**现象**：内存库跑得好好的，切 Postgres 后 `external_job` 外键违例。**根因**：子表在 stage_run 落库前就被写、且删除重插+CASCADE 冲突。**修复**：调整保存顺序 + Flyway 迁移软化外键。**收获**：无外键约束的内存模型迁到有约束的真库，会暴露被掩盖的时序假设。

### Bug 3：跨镜头人物不一致 / 免费图床挂了
> **现象**：视频里同一角色每个镜头长得不一样。**根因**：我一开始用"同 seed 独立出图"，flux 同 seed 不同 prompt 并不保证同一张脸。**修复**：改成 **i2v 末帧接力**（上一段末帧作下一段输入，像素级续接），人物一致性从 60 提到 98；期间还踩了公网图床（catbox/0x0）全挂，改用 uguu 托管帧。**收获**：一致性要靠"续接/参考"机制，不能靠模型随机性碰运气。

**其它可讲**：TTS 连打 12 次触发 429 → 退避重试；DashScope 图片模型/端点不匹配 → 定位到网关端点只认 wan2.6-t2i。
